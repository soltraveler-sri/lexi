## Theme of v1.5

If v1 is "the transforms iteration," v1.5 is **"the editing-room iteration."** Margin comments, agent passes, agent panels, in-line chat, and the second wave of transforms together turn lexi into a place where the user runs reviews, not just rewrites. The user's relationship with lexi shifts: in v1 they have a sharp set of tools; in v1.5 they have a *team in the margins*.

The hierarchy of importance in v1.5:

1. **Margin comments primitive** — the missing UI affordance that unlocks everything agentic. Every other v1.5 feature gets more interesting once margin comments exist.
2. **Agents Level 2 + Level 3** — the agent platform earns its keep. A single "Send to my editor panel" command runs three or four named agents in parallel, each posting threads in the margins, with a chair agent producing a synthesis.
3. **Doc-level transform expansion** — tone shift, audience swap, restructure, citation, glossary. The doc-level surface from v1 grows into a real toolbox.
4. **In-line chat** — a small, contained back-and-forth that progresses naturally from any in-line transform.
5. **Style Profile maturation** — per-doc overrides, anti-patterns, exemplar provenance.
6. **Journal expansions** — Notes grows up, Reflections dashboard ships its first real version.

## Cross-cutting decisions

- **Margin comments are the new universal output channel.** Once shipped, agents prefer comments over modal panels for non-rewrite output. The side-panel response flow from v1 doesn't go away (it's still right for whole-doc summaries), but per-passage feedback moves into the margins.
- **Agent panels are concurrent, not sequential.** Multiple agents read the same doc state in parallel. The chair runs after they all complete. Don't build a sequential pipeline pretending to be a panel.
- **Cache aggressively at the agent/persona level.** When an agent runs a 60-block pass on a doc, every block call shares the same `(persona_prompt + voice_profile)` system block. This is the canonical case for Anthropic ephemeral cache — make sure cache hit rates are visible in the usage dashboard.
- **In-line chat does not bleed into long-term memory.** The chat exists for the duration of the in-line strip session. When the strip closes, the transcript is discarded. What persists is *distilled insight* extracted into the corpus, not the chat itself.

## 1. Additional in-line transforms (restructure, citation, continue-within-selection)

### Goal

Round out the in-line transform shelf. Three new transforms operating on a selection.

### What ships

- **Restructure (in-line)** — keep the words, change the order/flow of the selected passage. Useful for a paragraph that has the right pieces but the wrong sequence. Returns the same content reorganized; voice profile composes; same streaming + 2-variant pattern as v1's transforms.
- **Citation (in-line)** — for a selected claim or paragraph, find supporting sources via web search (built on v1 feature 7), then propose 1–3 citations to insert. The output is *not* a rewrite of the selection's prose; it's a structured "here are sources, want me to insert footnote markers / inline links?" panel. Insertions are confirmed by the user. Each insertion is a small edit anchored to the existing selection — no rewriting of the surrounding prose unless the user explicitly opts into "rewrite to make the citation flow naturally."
- **Continue-within-selection** — given a selected partial sentence or partial paragraph, continue it in voice. Different from the v1.5 doc-level "continue from end of doc" pattern (which we are explicitly *not* shipping in v1.5; that's a Future consideration). This is a precise tool for the case where the user's mid-thought and wants the model to finish that thought, in voice, within the surrounding context.

### UX intent

- All three follow the established in-line strip pattern: bubble menu → transform → strip with streaming output → accept/edit/reject.
- Citation is the exception: its output is a structured suggestion, not a streaming rewrite, so it gets a small variant of the strip — a narrow panel showing the candidate citations with title, URL snippet, and "insert" / "dismiss" buttons.
- Restructure should default to a short reasoning prelude in the strip ("I moved [X] before [Y] so the cause-and-effect lands first"), or — better, less cluttered — show only the rewritten passage but let the user click an info chip to see the why. The latter is preferred.

### Per-shape prompting reminder

Restructure (in-line) and Restructure (doc, v1.5 feature 2) share a name. They are different prompts. The in-line version is constrained to the selection and its tight surrounding context; the doc version operates on the whole document and considers global flow.

### Architecture notes

- Restructure and Continue-within-selection extend the in-line `Transform` registry exactly like v1 transforms. Streaming + voice profile composition + 2 variants where it makes sense.
- Citation reuses `webSearch()` from v1 feature 7. The transform's `run()` calls websearch first, then optionally a model call to filter/summarize/match candidates to the claim, then renders the candidates in the citation panel.
- Citation insertion: a small TipTap operation that wraps the relevant text in a link mark or inserts a footnote-style marker. Pick one v1.5-tractable approach (probably inline `<a>` links with hover-card showing the source) and ship it.

### Guardrails

- No transform ever calls multiple models silently in a way that surprises the user on cost. Citation is the high-cost one (web search + an LLM call); show this in the strip.
- Continue-within-selection must respect the *type* of the selection. If the user selects a fragment of a sentence, the continuation should join that fragment, not start a new sentence.
- Restructure should not change facts or claims; only reorganize.

### Acceptance criteria

- Three new transforms in the bubble menu and command palette.
- Each works end-to-end with streaming and (where applicable) variants.
- Citation produces real, clickable, dated sources.
- All three log `style_events` and `usage_events` correctly.

## 2. Additional doc-level transforms (tone shift, audience swap, restructure, citation, glossary)

### Goal

Expand the doc-level transform surface from v1 (which shipped with one: SoC → Draft / Outline) to a real toolbox of whole-document operations.

### What ships

- **Tone shift (doc level)** — same tone-axis vocabulary as the in-line version (formal ↔ casual, hedged ↔ direct, cold ↔ warm). Acts on the whole document. Snapshot first; preview the result; replace on commit.
- **Audience swap (doc level)** — same audience set as the in-line version (senior leader, technical peer, generalist reader, friend), applied to the whole doc. The model is told this is a wholesale audience pivot, not a tone tweak.
- **Restructure (doc level)** — reorganize the entire doc's structure (sections, paragraph order, transitions) while keeping the substance.
- **Citation (doc level)** — walk the document, identify claims that warrant citations, propose sources for each. The user reviews and inserts in batch (a "review citations" surface that lists all proposed citations across the doc, each with an insert/dismiss action). This needs the maturation in feature 3 (web search + verification).
- **Glossary (doc level)** — two modes:
  - **Create**: when the doc has no glossary attached, the transform extracts key terms and proposes a glossary (term → short definition).
  - **Normalize**: when a glossary exists, the transform walks the doc and proposes capitalization / spelling / terminology fixes against the glossary's canonical forms.

### UX intent

- All five surface through the same doc-action button entry point established in v1. The doc-action menu grows from "one transform" to a small categorized menu (e.g. "Restructure / restate," "Citations / sources," "Glossary").
- Tone shift, audience swap, and restructure all use the same "snapshot → preview → replace or save-as-new-doc" pattern as v1's SoC → Draft.
- Citation (doc) is different: instead of replacing the doc, it produces a review queue of proposed citations the user accepts or rejects one by one. The doc only changes if/when the user accepts insertions.
- Glossary needs a real home: glossaries are doc-or-project-scoped; the user should be able to view, edit, and re-use glossaries. v1.5 default: glossaries are doc-scoped, stored as JSON on the document or in a small `glossaries` table keyed by `documentId`. Project-scoped glossaries are a small extension that can come later.

### Per-shape prompting

Already emphasized in v1, but it remains the most important non-negotiable here: the doc-level versions of these transforms have prompts authored fresh for the doc shape. Tone shift on a paragraph is a tactical edit; tone shift on a doc is an authorial choice. The model should be told different things, see different context, and have different output expectations.

### Architecture notes

- Each transform is a new `DocTransform` registration in the registry created in v1 feature 3.
- Costs: doc-level transforms can be expensive. Show the user the rough scope before running. For citation (doc), show *both* the doc word count and the estimated number of web searches.
- Glossary table:

  ```
  glossaries (id, user_id, document_id (nullable, doc-scoped if set), terms jsonb, updated_at)
  ```
  where `terms` is `[{term, definition, aliases?: string[]}, ...]`. Project-scoped glossaries (nullable `document_id`, future `project_id`) are a small extension to the same table.

### Guardrails

- Doc-level citation must not silently insert into the doc. Always a review queue.
- Glossary "normalize" mode must show the user what it's changing before making changes (preview / batch-confirm). Silent global find-replace by an AI model is a recipe for trust loss.
- Don't conflate "audience swap" and "tone shift" in the prompts. They overlap conceptually but the user picks them deliberately; honor their choice.

### Acceptance criteria

- All five new doc-level transforms work end-to-end.
- Cost preview is shown before running.
- Snapshots are taken before any destructive replacement.
- Citation and glossary use review queues, not silent inserts.
- Glossary can be created, viewed, and edited.

## 3. Web search maturation (citation pipeline + source verification)

### Goal

The v1 web search layer is barebones (one query, results back). v1.5 needs it to support real citation work: claim extraction → multi-query search → result filtering → source verification → insertion.

### What ships

- A `citationPipeline()` helper that, given a passage or doc:
  - Extracts factual claims (model call).
  - Generates targeted search queries per claim (model or heuristic).
  - Runs web searches in parallel (rate-limited).
  - Scores and ranks results per claim.
  - Optionally fetches the top result(s) and verifies the claim against the source content (a second model call).
- A small "source verification" subroutine: given a claim and a candidate source URL, have the model say "supports / contradicts / unrelated / unclear."
- Improved usage tracking: each pipeline stage is its own `usage_events` row so we can see where cost goes.

### UX intent

- The pipeline is invisible to the user. They see only the proposed citations in the citation transform's review queue, with confidence indicators ("verified against source" / "matched by snippet only").
- When a result is "contradicts," the citation panel surfaces this clearly — the user might learn their claim is wrong, which is itself valuable.

### Architecture notes

- Build the pipeline as composable steps, not as one big prompt. Each step is a tractable LLM call with a focused job. This is also where prompt caching pays off — the doc text is the same across many claim extractions.
- Cap the number of claims processed per run (e.g. 20). For longer docs, paginate.
- Web fetch (for source verification) must be sandboxed and time-bounded. Don't build a general-purpose fetcher; build something that returns a clean text dump of an HTML page, with a hard timeout and size cap.

### Guardrails

- Don't fabricate citations. If the pipeline can't find a credible source, return "no citation found" rather than a plausible-looking made-up one.
- Don't claim verification when only matching against a snippet. The "verified" label is reserved for cases where the actual page content was fetched and assessed.
- Be conservative about how many parallel web fetches run.

### Acceptance criteria

- Citation transforms (in-line and doc-level) use the matured pipeline.
- Each citation in the review queue carries a confidence indicator and source URL.
- "No citation found" is a normal, non-error result.

## 4. Margin comments primitive

### Goal

Ship the **margin comment** as a first-class document affordance: range-anchored threads attached to a passage. This is the missing UI piece that unlocks Agents Level 2 and 3.

### What ships

- A new `comment_threads` table (illustrative names):

  ```
  comment_threads (
    id, user_id, document_id,
    range_from int, range_to int,
    anchor_text text,                        -- snapshot of the text at thread creation
    status (open | resolved | dismissed),
    created_by (user | agent_id),
    summary text,                            -- optional one-liner shown when collapsed
    created_at, updated_at
  )

  comment_messages (
    id, thread_id, body_md text,
    author (user | agent_id),
    created_at
  )
  ```
- Editor extensions to:
  - Mark a range as commented (a subtle highlight + a margin-rail dot).
  - Open a thread in the right margin when the user clicks the dot or selects "Add comment" from the bubble menu / `Mod+Shift+M`.
  - Auto-anchor: if the user edits the surrounding text, the thread moves with the anchor (TipTap mark-based anchoring is the right approach; if a thread's anchor is fully deleted, the thread becomes "orphaned" and lives in a doc-level "orphaned threads" panel).
- A right-margin gutter showing collapsed thread badges. Click expands to a thread card.
- Thread actions: reply, resolve, dismiss, link to source agent (if agent-authored).

### UX intent

This is the section where UI care matters most. Three constraints:

- **Calm by default.** A doc with 30 open threads should not look chaotic. Collapsed threads in the margin rail should be small, restrained, possibly color-coded only when an agent author is involved (so the user can quickly distinguish their own thoughts from agent feedback).
- **Visible enough to use.** Threads should not be so small that the user forgets they exist. The right rail should always show whether there are threads in the current viewport.
- **Tactile.** Hovering the highlighted text should surface the thread; clicking the rail dot should expand it; the cursor should always know where the thread anchors.

Specific design decisions to make during implementation (don't pin them down here):

- Whether the right rail is always visible or appears on hover.
- Whether agent threads are color-coded by agent or by role (editor / critic / etc.).
- How resolved threads are represented (faded, hidden by default, in a "resolved" tab).

The principle: **the user should be able to look at a doc with comments and immediately read it as a doc, with the comments as ambient context.** Comments should never grab attention away from the prose.

### Architecture notes

- TipTap mark for anchoring. Existing `ExemplarMark` is a good reference.
- Right-margin rail as a sibling-rendered React layer that aligns with the editor (use editor positions to compute Y coords). Not in TipTap itself.
- Thread CRUD endpoints under `/api/documents/[id]/threads`.
- Author union type (`user` | `agent_id`) handled at the application layer.

### Guardrails

- Don't reinvent Google Docs comments wholesale. We need range-anchored threads, replies, and resolution. We do not need notifications, mentions, or sharing in v1.5.
- Don't let agent-authored threads be silently mutated by other agent runs. Each thread is owned by its creator (the user or the specific agent).
- Don't store the full anchor text more than once — the snapshot is for display when the underlying prose has shifted; the live anchor is positional.

### Acceptance criteria

- The user can add a comment to a selection, see it in the margin, reply to it, and resolve it.
- Comments survive ordinary edits in the surrounding text. Comments whose anchor is fully deleted move to "orphaned."
- Comments are persisted and reload with the document.
- The right margin rail is calm and discoverable.

## 5. Agents Level 2 — block-by-block document passes

### Goal

An agent can run a **pass** over a whole document, walking it block by block, deciding per-block whether to comment, suggest, skip, or escalate. Output lands as margin comment threads, not as side-panel responses.

### What ships

- A new endpoint, `/api/ai/agents/[id]/pass`, that:
  - Walks the document's TipTap JSON block by block.
  - For each block, calls the agent (with cached system block: persona + optional voice profile).
  - Receives a structured per-block response: `{action: 'comment' | 'suggest_rewrite' | 'skip', body: string, suggested_replacement?: string}`.
  - Creates a `comment_thread` for each commented or rewritten block.
  - Streams progress back to the client (a "running pass on block 3 of 27" indicator in the UI).
- A "Run [agent] on doc" entry point in the agent's surface that triggers a pass.
- Per-pass cost preview: word count × per-block cost estimate.
- A "passes" history view per doc — what runs were performed, by which agents, when, with what outcomes.

### UX intent

- Running a pass should feel like the agent is reading the doc. A subtle progress indicator is fine; a heavyweight progress modal is not. The user should be able to keep editing while a pass runs (with the proviso that comments anchor against current positions when they land).
- When the pass completes, the user gets a small notification ("Cold Editor finished — 8 comments left") and can scan the rail to see them.
- Pass results are batches — the user might "review all from this pass" and walk through them, accepting/dismissing as they go.

### Architecture notes

- Cache the system block (`persona_prompt + voice_profile`) for the duration of the pass. Anthropic ephemeral cache is the right tool. Cache key includes agent ID + voice profile compiled timestamp so a profile update invalidates.
- Block-by-block calls run sequentially with bounded concurrency (e.g. 3 parallel) to avoid rate-limit cliff edges.
- A pass writes a row to a new `agent_runs` table:

  ```
  agent_runs (id, agent_id, document_id, started_at, finished_at, status, blocks_processed, comments_created, error)
  ```

### Guardrails

- A pass cannot run on a doc that's currently being passed by the same agent (no double passes). User should be able to run different agents in parallel; same agent only one at a time per doc.
- Per-pass cost cap. Show the estimate; warn at >$X (configurable per deployment); refuse at >$Y unless explicitly confirmed.
- Comments authored by an agent must be tagged with `created_by = agent_id` so the rail can render them distinguishably and the user can filter them.
- An interrupted pass (network failure, user cancel) leaves the comments it created so far. The pass row is marked `partial`.

### Acceptance criteria

- The user can trigger a pass from any agent's surface.
- The pass walks the doc and creates comment threads.
- Cost preview is shown before running.
- The user can cancel mid-pass.
- The `agent_runs` row reflects what happened.

## 6. Agents Level 3 — agent panels and chair-of-panel synthesis

### Goal

Run multiple agents on the same doc in parallel and produce a synthesized "what to do next" summary. This is when the agent platform stops being a power-user feature and becomes a meaningful daily workflow.

### What ships

- A new concept, **panels**: a user-defined named set of agents (e.g. "Pre-publish review = [Cold Editor, Continuity Editor, Skeptic, Generous Reader]"). Stored as:

  ```
  panels (id, user_id, name, agent_ids uuid[], chair_agent_id (nullable), created_at, updated_at)
  ```
- Panel management UI in Journal → Style Guide → Agents (or a dedicated tab there).
- A "Send to panel [Pre-publish review]" doc-action that:
  - Runs each agent in the panel as a Level 2 pass (in parallel, sharing the same doc snapshot).
  - When all complete, runs the chair (if defined) with the consolidated set of comment threads as input.
  - The chair produces a synthesis: a short top-level summary, the highest-leverage suggestions, and explicit notes on conflicts ("Cold Editor wants to cut paragraph 3; Generous Reader thinks it's the strongest moment").
  - The synthesis lands as a special doc-level thread (or a side-panel review summary), not in the rail.
- Visual treatment in the rail to distinguish multiple agents' threads (color or icon per agent).
- A "panel run" row in `agent_runs` (or a new `panel_runs` table) with FK to the per-agent runs.

### UX intent

- A panel run is the moment the user feels the editing room come together. The progress indicator should show all agents' progress at a glance.
- The chair's synthesis is the *first thing the user sees* after a panel run — a clean, readable summary up top, with the underlying threads available for deeper inspection.
- Conflicts surfaced by the chair should be presented as honest disagreements, not papered over.

### Architecture notes

- Panel run = N parallel `agent_runs` + 1 chair run.
- The chair receives: the doc text + all comment threads created by the panel + the user-facing summaries each agent left at thread level. The chair's prompt asks for: a 2–4 sentence top-level read, 3–6 highest-leverage actions, and any explicit conflicts.
- The chair is itself an agent in `agents`. Its `output_kind` is something like `synthesis`. We may want to add this as a new `output_kind` value in v1.5.
- A "default chair" comes pre-seeded so users don't have to build one.

### Guardrails

- Panels are capped at 5 agents in v1.5 (sanity limit; can grow).
- Panel runs share a single doc snapshot — every agent reads the same state. Otherwise comments anchor against shifting text and the chair's synthesis is incoherent.
- The chair never modifies the doc directly. It comments. v1.5 does not give agents write access to the doc body itself outside of the rewrite-strip flow on selections.
- Panel runs that error partway show what completed and what didn't, instead of failing wholesale.

### Acceptance criteria

- The user can define a panel with 2–5 agents and an optional chair.
- "Send to panel" runs all agents in parallel and produces a chair synthesis.
- The synthesis is presented prominently after the run.
- All per-agent comments are accessible in the rail.

## 7. Pre-built agent persona starter set (expanded)

### Goal

Beyond the v1 starter pack, v1.5 ships a richer library of starter personas tuned to the v1.5 capabilities (passes, panels). Reduce the "blank canvas" problem for the average user.

### What ships

Add starters covering:

- **The Continuity Editor** — silent unless something contradicts an earlier passage in the doc.
- **The Big-Picture Critic** — looks for the through-line, asks "what is this piece really about?"
- **The Sentence-Level Copy** — pure mechanics: rhythm, redundancy, parallelism. (Different from Sentence Surgeon in v1; Sentence Surgeon rewrites, this one comments.)
- **The Voice Guard** — runs cheaply on commit (in v1.5 this is still a manual run; v2/Future makes it automatic). Flags drift from the voice profile only.
- **The Translator** — re-renders the doc for a different audience type. Already similar to Audience Swap (transform) — but as an agent with persistent persona, more flexible.
- **The Default Chair** — for panel synthesis.
- A few "pre-built panel" starters: "Pre-publish review," "Quick second opinion," "Continuity check."

### UX intent

- Starters are clearly labelled. Users feel zero friction in cloning, tweaking, or replacing them.
- The starter library is browsable in the agent management surface. Each starter has a one-paragraph description of its tasted, sample output, and recommended use cases.

### Architecture notes

- Starters are seed data — committed in code, not in the DB. The user clones a starter into their own `agents` row.
- Pre-built panels are similarly seed data; cloning a panel clones each of its agents (or references them, depending on what's cleaner).

### Acceptance criteria

- The starter library is browsable.
- Cloning works for both individual agents and panels.

## 8. In-line chat (session-scoped, distillation-only persistence)

### Goal

Let the user have a short back-and-forth with the model from inside the in-line strip, without inventing a separate chat surface. Chat is scoped to the strip session and ends when the strip closes. Insights distilled from the chat may persist; the transcript itself does not.

### What ships

- An expandable "ask a follow-up" affordance in the in-line strip (visible after a transform or rewrite suggestion has rendered). The user types a question or a tweak ("can you make this less formal?", "what if I cut the second sentence?"). The model responds, optionally producing a new candidate suggestion.
- The chat history is visible only while the strip is open. Closing the strip discards the transcript.
- A small "extract insights" pass runs at strip close (asynchronously). It looks at the chat for any patterns useful to the corpus (e.g. "user kept asking for less formal tone"), and proposes them as candidate preferences via the same review queue as feature 11 / v1 feature 8. Nothing auto-commits.

### UX intent

- The chat must feel like a natural extension of the same interaction the user is already in. Hitting enter on a follow-up should keep the strip the visual focus.
- The chat must NOT feel like a chatbot. No avatars, no persistent threads, no "good question!" preamble. Tight, working back-and-forth.
- The strip's existing "accept / reject" affordances continue to operate on the latest candidate the model has produced (the chat may have iterated through several).
- When the strip closes, the user should feel the conversation ended cleanly. No "would you like to save this chat?" friction.

### Architecture notes

- Chat state lives in the existing `RewriteSession` (in `RewriteStripProvider`) — extend it with a `messages` array (alternating user/assistant). Cleared on close.
- Each turn is a streaming model call with the full chat history + the original passage + the voice profile.
- "Extract insights" is a small async task on close. Cap how often it runs (e.g. only if the chat had ≥ 3 user turns and produced a different candidate from the original suggestion).

### Guardrails

- **No persistence of chat transcripts.** This is a non-negotiable privacy and simplicity boundary.
- No long context windows. The chat caps at, say, 8 turns; after that, the model gets a polite "let me restart with a fresh suggestion" treatment.
- Don't surface chat in agents in v1.5. Agents are doc-pass-shaped; chat is selection-strip-shaped. These remain distinct.

### Acceptance criteria

- The user can have a multi-turn refinement chat inside the in-line strip.
- Closing the strip discards the chat.
- Insights candidate-extraction runs and lands proposals in the review queue.

## 9. Style Profile expansions

### Goal

Three small but high-leverage upgrades to the Voice Profile system.

### What ships

- **Per-document overrides.** A doc can pin a different `voice_context`, attach extra preferences, or exclude specific exemplars. Stored as a small JSON column on `documents` (e.g. `voice_overrides jsonb`). UI lives in the doc settings (a "Voice settings for this doc" panel reachable from the doc chrome).
- **Anti-patterns / "not me" list.** A new `style_anti_patterns` table:

  ```
  style_anti_patterns (id, user_id, content text, created_at, updated_at)
  ```
  Composed into the voice profile as a "do NOT do these things" block. UI lives alongside `style_preferences` in Journal → Style Guide.
- **Exemplar provenance.** Extend `exemplars` with optional `source_url`, `source_title`, `source_author`, `is_self_authored boolean default true`. Allows the user to mark exemplars from writers they admire (not just their own writing). The voice profile compiler treats self-authored and external exemplars differently: external ones are presented as "writers I'm influenced by," not as "my voice."

### UX intent

- Per-doc voice settings should feel like a quiet override, not a re-onboarding.
- Anti-patterns should feel like writing rules ("avoid em-dashes," "never start a sentence with 'However,'") not like banned-words lists.
- Exemplar provenance should be a couple of optional fields the user can fill in, not a required form.

### Architecture notes

- The voice profile compiler grows a new section for anti-patterns (when present) and a new section for "writers I'm influenced by" (when external exemplars exist).
- Per-doc overrides flow into the resolver via a new `compileVoiceProfile()` parameter (`overrides?: VoiceOverrides`).

### Guardrails

- Don't over-invest in per-doc UI. Most users will never touch it; it's there for the cases where they need it.
- Exemplar provenance is opt-in. Backfill default is "self-authored, no source data" for existing rows.
- Anti-patterns shouldn't be auto-extracted from edits in v1.5. Manual entry only. (Auto-extraction of anti-patterns is a Future idea once we have enough rejection signal.)

### Acceptance criteria

- A doc can have per-doc voice settings that take effect when AI runs on it.
- Anti-patterns can be added, edited, deleted, and are reflected in the voice profile.
- Exemplars can be marked with source metadata.

## 10. Journal expansions (Notes maturation, Reflections dashboard v1)

### Goal

Promote the Notes panel from "single-doc notepad" to "lightweight, useful notes surface." Ship the first real version of the Reflections dashboard.

### What ships

- **Notes maturation.**
  - Multiple notes (not just one). A small list with create / rename / delete.
  - Tags on notes.
  - Search across notes.
  - An opt-in setting: "Include notes as context for the voice profile" — when enabled, the compiler reads the notes' content and weaves relevant excerpts into the profile under a "user's meta-thoughts on their writing" section. Off by default.
- **Reflections dashboard v1.**
  - **Output stats** — words written this week / month / quarter, by document type.
  - **Edit signal** — top edit tags, top transform IDs (from `style_events.editTags` and `usage_events.transformId`).
  - **AI relationship** — accept / edit / reject ratios on AI suggestions over time. This is the most honest view of where AI is and isn't earning its keep for the user.
  - **Voice drift** — surface the most-rewritten openings and most-rewritten phrases, drawn from `style_events`.

### UX intent

- Reflections is *not* a productivity dashboard. It's a writing journal. The numbers should feel like reflections, not KPIs.
- Notes should feel like a journal, not Evernote. A small list, a clean editor, no "spaces" or "stacks."
- The opt-in to use notes for voice profile is a one-line toggle with a clear explanation; it's not opt-in-by-default and we don't pester users to enable it.

### Architecture notes

- Schema for notes:

  ```
  notes (id, user_id, title text, content text, tags text[], updated_at)
  ```
- Reflections queries are read-only aggregations against existing tables. None of the dashboard requires new state.

### Guardrails

- Don't gamify Reflections. No streaks, no goals, no "you wrote 30% less this week" guilt.
- Don't pull notes into the voice profile silently. The opt-in is explicit and reversible.

### Acceptance criteria

- Multiple notes work end-to-end.
- The opt-in toggle composes notes into the voice profile correctly.
- The Reflections dashboard renders the four sections above with real data.

## 11. Auto-extracted preferences — periodic background job

### Goal

Wrap v1's manual "Analyze my edits" flow in a periodic background job (default weekly cadence), with smarter pattern detection.

### What ships

- A scheduled task (Vercel cron, or whatever the deploy target supports) that runs the analysis weekly per active user.
- Smarter prompts: cluster recent edits before analysis; deduplicate against prior accepted/rejected candidates; detect when an existing preference has become stale (e.g. user has been ignoring it consistently).
- Notifications: a small in-app indicator in Journal → Style Guide when new candidates are awaiting review. No email in v1.5.

### UX intent

- The user opens the journal one day and sees: "I noticed three new patterns in your last week of writing." Click to review. That's the whole product.
- No nagging. No badges in the workspace. The notification is restricted to Journal → Style Guide.

### Architecture notes

- Persist the candidate-review history so deduplication works across runs:

  ```
  preference_candidates (id, user_id, content text, status (proposed | accepted | rejected | superseded), evidence_event_ids uuid[], created_at, decided_at)
  ```
- The job reads recent `style_events` since the last successful run, generates candidates, dedupes against `preference_candidates` and `style_preferences.content`, writes new `proposed` rows.

### Guardrails

- Cap candidates per run at 5–7.
- Don't propose anti-patterns automatically yet (still manual entry per feature 9).
- Respect a user opt-out. A setting "Don't auto-analyze my edits" is mandatory.

### Acceptance criteria

- The weekly job runs and produces candidates.
- Candidates dedupe correctly against history.
- Users can opt out.

## 12. Variants count revisit (only if v1's 2-variant proves under-served)

### Goal

If v1 ships 2 variants and feedback / data indicates 3 would be net-positive, expand. Otherwise leave it alone.

### Decision criteria

- Look at `style_events`: how often does the user edit the accepted variant before commit?
- Look at qualitative feedback: do users ask for more options?
- If yes: bump to 3 in select renderer modes (overlay modal can stack 3 cleanly; inline strip can't, so it stays at 2).
- If no: do not change the count. More options is not better by default.

### Guardrails

- Do not bump variants count silently. This is a deliberate, evidence-driven decision.
