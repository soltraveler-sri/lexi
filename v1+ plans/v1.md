## Theme of v1

v1 is the iteration where lexi stops being "a Rewrite Strip with a Style Profile" and starts being a **transforms-first writing surface with a first-class Voice Profile**. Two new product shapes appear: **doc-level transforms** (acting on the whole document) sit beside the existing **in-line transforms** (acting on a selection); and **agent personas** make their first appearance as composable layers on top of the voice profile.

The hierarchy of importance in v1:

1. **Stream of Consciousness → Draft / Outline** is the single most important feature in this iteration. A user opening a blank document and dumping a rant should be able to one-click structure it into either a draft or an outline, in their voice. This needs to be discoverable, fast, and obviously the headline of the new doc surface.
2. **In-line transform shelf** widens the daily-driver toolkit so the user has more reasons to live in lexi.
3. **Voice Profile** gets two upgrades: a guided creation walkthrough for new users, and a manual "Analyze my edits" pattern-extraction action for existing users.
4. **Agents Level 1** establishes the agent primitive without overreaching — a single persona running on a selection or a doc, composed with the voice profile, output via existing renderers (no new comment surface yet).
5. The **Journal surface** restructures the top-level IA so reflection, context, and notes have a permanent home.

## Cross-cutting decisions

These apply across every v1 feature.

- **Two registries, two shapes.** In-line transforms (existing `Transform` interface in `src/lib/transforms/`) operate on a selection. Doc-level transforms (new) operate on the whole document. They share concepts (voice profile, document type, voice context, usage events) but have separate types, separate routes, and separate UI surfaces. Resist the urge to unify them into one abstraction — the UX shape is genuinely different and forcing one shape will cripple both.
- **Per-shape prompting.** A transform with the same name (e.g. tone shift) is a different feature when applied to a selection vs. the whole doc. The prompts must be authored separately. Never share a single prompt template across the two shapes.
- **Streaming everywhere it makes sense.** The provider already exposes `stream()`; route handlers should pipe it to the client. The existing rewrite endpoint should be migrated to streaming as part of feature 1.
- **Voice profile composition is the default.** Every AI-facing v1 feature uses the user's compiled voice profile as the system context unless the user explicitly bypasses it.
- **Cost transparency for doc-level operations.** Doc-level transforms can be 10–100× the cost of an in-line rewrite. Show the user a rough order of magnitude before running ("This will read your full document — about ~2,500 words. Continue?"), and snapshot the doc before any destructive replacement.

## 1. Streaming rewrites + 2-variant Rewrite Strip

### Goal

Make the existing Rewrite Strip feel alive (typewriter-render the suggestion as it streams) and give the user a built-in choice between two variants instead of one. Variants are the cheapest way to give the user agency over taste without forcing them to re-prompt.

### What ships

- `/api/ai/rewrite` migrated to streaming. The route should stream to the client; the controller should render incrementally.
- The Rewrite Strip renders **two variants** side by side (or stacked, depending on renderer mode). User picks one with `Cmd+1` / `Cmd+2`, with `Tab` to focus, with click, or by editing one of them.
- The unpicked variant is logged as `ai_suggestion_rejected` (so we capture that signal in the corpus for free).
- The picked variant flows through the existing commit path (`ai_suggestion_accepted` if unedited, `ai_suggestion_edited` if the user changed it before committing).

### UX intent

- The strip should *feel like a fast assistant typing alongside you*, not like a form submission round-trip.
- Variants should feel like "two takes from the same writer" — close enough to both be plausible, different enough to matter. Prompt them with a brief instruction to vary axis (e.g. one tighter, one warmer), not free-for-all randomness.
- Two variants is the right number for v1. Three is too much real estate, especially in inline mode. We may revisit later.

### Architecture notes

- Anthropic and OpenAI providers both already implement `stream()`. The rewrite endpoint should call `stream()` and forward the chunks via Server-Sent Events or a streaming `Response`.
- Rendering two variants in inline mode requires the renderer to host two side-by-side `RewriteInput`-shaped nodes. Keep the locked-original node identical; duplicate the input node with an index.
- The variants should be requested **in parallel** as two streams (rather than one call asking for two outputs). This is faster and the rejected one is naturally usable for the rejected-variant `style_event`.
- For prompt diversity: pass two slightly different short instruction suffixes ("be tighter and more direct" vs. "be more conversational and warmer"), or use a small temperature spread (0.5 vs 0.9) — pick whichever feels more on-voice in testing. Avoid asking the model to "give me two different versions" in a single call; the model collapses to a hedged middle.

### Guardrails

- Do not show the variant UI when AI is unavailable — the strip falls back to the existing manual rewrite flow.
- Do not treat the rejected variant as a hard signal; only log it as `ai_suggestion_rejected`. The corpus prompt compiler already handles this event type.
- Do not increase the rate-limit budget per rewrite implicitly. Two variants = two API calls = two `usage_events` rows. The user should see this in their usage dashboard.
- Do not break the existing renderer-mode contract. The variant strip must work in `inline_strip`, `side_panel`, and `overlay_modal` (the latter two can stack rather than side-by-side).

### Acceptance criteria

- Selecting text, hitting `Mod+R`, and pressing "Suggest with AI" returns two suggestions, each rendering progressively as the model streams.
- `Cmd+1` / `Cmd+2` pick a variant; `Cmd+Enter` commits the focused variant; `Esc` cancels.
- The unpicked variant is recorded as `ai_suggestion_rejected` with the same surrounding context as the accepted one.
- All renderer modes work.

## 2. In-line transform shelf (tighten/loosen, tone shift, audience swap)

### Goal

Take the existing single-transform Rewrite Strip and turn it into a small shelf of in-line transforms the user can apply to a selection. This is where the daily-driver value compounds: the user has a real toolbox, not one button.

### What ships

Three new transforms, in addition to the existing rewrite:

- **Tighten / Loosen** — the same transform with a direction parameter. Tighten by ~20–30%; loosen by ~20–30% (default ratios; the user can nudge later via a stepper if we add one). Preserves meaning, voice, and continuity.
- **Tone shift** — moves the selection along a tone axis. v1 ships a small fixed set of axes: formal ↔ casual, hedged ↔ direct, cold ↔ warm. The user picks the axis and the direction.
- **Audience swap** — rewrites the selection for a different audience. v1 ships a small fixed set: senior leader, technical peer, generalist reader, friend. User picks the target.

Each transform:

- Lives as its own `Transform` registration in `src/lib/transforms/`.
- Hits a transform-specific endpoint (or a single `/api/ai/transform/[id]` endpoint with per-transform handlers — your call). Per-transform endpoints are nicer for cost analytics.
- Shows up in the bubble menu when text is selected, and in the command palette.
- Logs a `style_event` on accept with `transformId` set so we can analyze adoption per transform.
- Stamps `usage_events.transformId` for cost analytics.

### UX intent

- The bubble menu, when text is selected, should feel like a small, calm shelf: the transforms appear, the user picks one, the strip opens with a streaming suggestion already underway. Selection → transform → result should be at most two clicks.
- Hotkeys for the most-used: `Mod+R` rewrite (existing), `Mod+Shift+T` tighten, `Mod+Alt+T` tone shift, etc. Don't over-claim hotkey real estate; let the bubble menu carry most discovery.
- The "direction / axis / audience" parameter for non-rewrite transforms must be picked *before* the model runs — never present a result first and let the user retroactively tag what it was. The right shape is: select → click "Tone shift" in bubble menu → mini submenu to pick axis + direction → strip opens with suggestion.
- Variants from feature 1 apply here too: every transform can return two variants. Default on for rewrite and tighten/loosen; default off for tone shift and audience swap (where the parameter already does the work of disambiguating).

### Architecture notes

- The existing `Transform` interface already has `id`, `name`, `hotkey`, `requiresSelection`. Extend with optional `parameters` (ordered list of param descriptors) and a UI hook so the bubble menu submenu can render parameter pickers automatically.
- Each transform's prompt construction lives next to the transform definition. Reuse the voice-profile compiler as the system block. Pass the parameter values into the user prompt.
- **Per-shape prompting** is critical: these transforms also exist as doc-level transforms in v1.5; the in-line versions must be authored to act on the selection in the surrounding context, never to act on the whole doc. The selection-shaped prompt should always include a few hundred chars of preceding/following context for continuity.

### Guardrails

- Do not unify all transforms into a single "transform with a free-text instruction" UI. The point of having distinct transforms is that the prompts are tuned and the UX surface is constrained.
- Do not let tone shift and audience swap accept arbitrary user-typed parameters in v1. Fixed sets only. Free text is a v1.5 conversation if it earns its place.
- Do not leak transform-specific instructions into the voice profile. The voice profile should remain task-agnostic.

### Acceptance criteria

- Bubble menu shows the four transforms (rewrite, tighten/loosen, tone shift, audience swap) when a selection is active.
- Each transform supports streaming + 2 variants per the rules in feature 1.
- Each accepted transform writes a `style_event` with `transformId` set.
- Each transform's `usage_events` row carries the `transformId`.

## 3. Doc-level transforms — new shape + first transform (Stream of Consciousness → Draft / Outline)

> This is the most important v1 feature. Read it carefully. The UX intent matters more than the architecture.

### Goal

Introduce a second transform shape: **transforms that act on the whole document.** The first one — and arguably the headline feature of the entire app at this stage — is "I have a stream of consciousness in this doc; turn it into a draft (or outline) in my voice." The flow is: open a blank doc, dump a rant, click one button, get a structured first draft. This is the moment lexi becomes irreplaceable for early-stage thinking-into-writing.

### What ships

- A new transform shape: **doc-level transforms** — registered separately from in-line transforms, surfaced in a dedicated UI, and routed through a separate API path (`/api/ai/doc-transform/[id]` or similar).
- The first doc-level transform: **Stream of Consciousness → Draft / Outline**, with a toggle: target = `draft` (default) or `outline`.
- A doc-level transform UI surface that is:
  - **Prominent and inviting on a blank or near-empty document** (a centered card or panel offering the SoC → Draft transform with the toggle, and a quiet secondary "or just start writing" affordance).
  - **Available but unobtrusive on populated documents** (a doc-level action button in the document toolbar / chrome, plus command-palette entries).
- A snapshot is taken before any doc-level transform runs (we already have `document_snapshots` — use it).
- A confirmation step that shows the user the rough cost/scope (e.g. "This will read your full document — ~1,200 words.") before committing the API call.
- Streaming render of the result, ideally into a side-by-side preview vs. the original; user picks "Replace" or "Discard" once the stream completes (or while it's streaming, if they decide early).

### UX intent

This is the section where being precise matters most. Three scenarios drive the design:

**Scenario A — blank doc, user has a rant in their head.**
The user creates a new doc. They see a calm, centered card or pane that says something like "Stream of consciousness → [Draft / Outline]. Drop your messy thoughts in below; lexi will structure them in your voice." There's a toggle between Draft and Outline. There's a large, inviting input area (or it cleanly indicates: just start typing in the doc, then press the button).

The user types or pastes their stream. They click the button. The doc-level transform fires. The original is snapshotted. The result streams into the document, replacing the stream of consciousness with the structured draft or outline.

**Scenario B — partially-written doc, user wants to restructure.**
The user is mid-doc and decides what they have is too messy. They use a doc-level action (in the toolbar / command palette / a "doc actions" dropdown) to invoke "Stream of Consciousness → Draft." Same flow, but the entry point is a button rather than a centered card.

**Scenario C — finished draft, user wants an outline of it.**
Same entry point as B; toggle to Outline; the transform produces a structured outline of the existing draft. The original is preserved via snapshot. The user can either replace the doc with the outline (creating an "outline of" version) or keep the outline as a sidecar — actually, simplest behavior for v1: the result *replaces* the doc, the original is recoverable from snapshots, and the user can also explicitly choose "Save outline as new doc" before committing the replacement. The "save as new doc" path uses the existing `sourceDocumentId` field.

**Visual / interaction principles:**

- The blank-doc CTA should *feel like an offer, not a wall*. The user must always have an obvious "no thanks, I'll just write" path.
- The CTA should disappear as soon as the user starts writing meaningfully — say after 50 words — so it doesn't squat on the doc.
- The doc-level action button (for non-blank docs) should live in the document chrome (top-right of the doc, near the title / metadata), not in the bubble menu. The bubble menu is for in-line transforms; doc-level transforms have their own surface.
- The result preview should make it clear that the original will be replaced — the user should never feel like they accidentally lost their stream of consciousness.

### Per-shape prompting (critical)

The same transform name often exists in both shapes. The prompts MUST be different.

For SoC → Draft (doc level):
- The model is told: this is a stream of consciousness. The structure is loose, the ideas may be repeated, contradictions may exist. Pull out the through-line; make decisions where the user vacillated; produce a coherent draft in their voice.
- The model is fed the user's voice profile, the document type (blog post, work doc, fiction, etc.), and the voice context.
- The output is a finished draft, not commentary. No preamble, no "here's your draft," no headers about the input.

For SoC → Outline (doc level):
- Similar input handling, but the output is a hierarchical outline (top-level sections, sub-bullets) — formatted as TipTap-compatible structure (we'll need to render the model's markdown-ish output back into the editor's JSON).
- The model is told the user will write the actual draft *from* the outline; the outline should be specific enough to write from but not so detailed that it duplicates the eventual draft.

For each subsequent doc-level transform (in v1.5), the prompt is again authored fresh for that shape. **Never share prompts across shapes for the same conceptual transform.**

### Architecture notes

- New types in `src/lib/transforms/doc/types.ts` (e.g.):
  - `DocTransform` interface: `id`, `name`, `description`, `parameters`, `requiresContent: boolean` (some, like SoC → Draft, can run on near-empty docs; others, like glossary, need content).
  - `DocTransformContext`: `userId`, `documentId`, `fullDocumentJson`, `documentType`, `voiceContext`, `parameters`.
  - `DocTransformResult`: either a `replacement` (full new doc content as TipTap JSON) or a `sidecar` (new doc to create with `sourceDocumentId` set), plus an optional `summary` for telling the user what changed.
- New registry parallel to the in-line one: `src/lib/transforms/doc/registry.ts`.
- New route under `src/app/api/ai/doc-transform/[id]/route.ts`. Streams responses where possible. Stamps `usage_events.transformId` with `doc:<id>` (or similar prefix) so we can split in-line and doc-level usage cleanly in analytics.
- New UI surface: a `DocTransformPanel` component used in two modes:
  - **Inline-on-doc mode** — rendered in the document body when the doc is empty/near-empty.
  - **Action-button mode** — opened from the doc chrome. A modal or side panel.
- Snapshot before run via `document_snapshots`.
- Output rendering: streaming markdown-ish text needs to be converted into TipTap JSON. Use a small, well-tested converter (we already have `tipTapToMarkdown`; we need the inverse path for outline mode at minimum). For draft mode, the output is mostly paragraphs and headings — manageable.

### Guardrails

- **No silent destruction.** Doc-level transforms always snapshot first.
- **No hidden cost.** Always tell the user the rough scope (word count, cost order-of-magnitude).
- **No bypass of voice profile.** Use the heavy-tier voice profile for doc-level transforms by default. This is the surface where prompt caching pays off most — the same compiled voice profile gets reused across the user's frequent doc-level operations.
- **No conflating the two shapes.** A user invoking "Tone shift" on a selection (in-line) must never accidentally invoke it on the whole doc, or vice versa. The two surfaces should not visually share affordances.
- **No swallowing the user's content.** If the model returns nothing, an error, or content that's clearly not a valid draft/outline (e.g. it returned a refusal, or commentary), do not replace the doc. Show the error and leave the original intact.
- **No persistent CTA squatting.** The blank-doc CTA disappears once the user has written meaningfully (suggested threshold: 50 words). It can be re-summoned via the doc-action button.

### Acceptance criteria

- A new blank doc shows the "Stream of consciousness → Draft / Outline" CTA prominently, with a toggle between Draft (default) and Outline.
- The CTA disappears once the doc has >50 words.
- A doc-level action button is available in the doc chrome on any doc, exposing the same transform.
- Running the transform snapshots the doc, streams the result, and on commit replaces the doc content with the result.
- The user can also choose "Save as new doc" from the result preview, which creates a child document with `sourceDocumentId` set.
- A `usage_events` row is written with `transformId = "doc:soc_to_draft"` (or similar).
- The original is recoverable via snapshots.

## 4. Voice Profile onboarding walkthrough (brief / deep)

### Goal

Give new users — especially non-power users — a guided way to bootstrap a voice profile without writing a prompt themselves. The walkthrough is a model-led conversation that asks the user the right questions and turns the answers into a well-structured `style_preferences.content` block.

### What ships

- A new onboarding surface (and a re-runnable "Set up my voice profile" entry point in the Journal → Style Guide panel) that offers three options:
  - **Brief walkthrough** — a small, fixed set of questions (5–8). Takes a couple of minutes.
  - **Deep walkthrough** — a longer, branching set of questions (15–25, branching on prior answers). Takes 15–20 minutes.
  - **Skip / write my own** — links to the manual `style_preferences` editor we already have, with an explanation of what makes a good prompt.
- Both walkthroughs are model-driven. The questions are generated by a model with a system prompt explaining the purpose (compile a voice profile from the user's answers). The user types short answers; the model can dynamically follow up if an answer is too vague (in deep mode).
- At the end, the model **drafts** the voice profile prompt from the user's answers. The user reviews it and either accepts it as their `style_preferences.content`, edits it inline, or re-runs the walkthrough.

### UX intent

- The walkthrough should feel like a *short conversation with someone who's getting to know your writing*, not like a tax form.
- The model's tone in the walkthrough is warm but efficient. Each question should justify itself — a good walkthrough question signals to the user why it matters ("Some writers love long, layered sentences; others swear by short ones. Which feels more like you?").
- The brief walkthrough must produce a usable voice profile. The deep walkthrough produces a richer one. Both are better than nothing.
- The end state — the drafted voice profile — should be presented in the same editor / surface the user will see when reviewing their style preferences later, so they understand what they're agreeing to.

### Important calibration on what this walkthrough is for

The user's voice profile in lexi is currently three parts: `style_preferences` (freeform notes), `exemplars` (marked-up passages of their own writing), and `style_events` (rewrite before/after pairs accumulated over time). The walkthrough only writes to **`style_preferences`**. It does not invent exemplars and it does not pre-seed style events — those continue to grow naturally from the user's actual writing.

The walkthrough is most valuable for:
- New users with no writing history yet (the "cold start" problem).
- Existing users who know they want a voice but haven't articulated it in writing yet.
- Users who would write a bad prompt themselves and would benefit from structured questions producing a well-shaped result.

### Architecture notes

- A small new surface (route, e.g. `/journal/style-guide/walkthrough`) that owns the walkthrough state machine: question, answer, next question, etc.
- The model interaction can either be a single multi-turn chat with the same model session, or a series of stateless calls each given the prior Q/A as context. v1 default: stateless calls with the running Q/A history passed each time. Simpler, easier to debug, easier to retry.
- A separate "compose final prompt" call at the end with all the answers — the system prompt for this call is "you are writing a voice profile prompt for the user; here are their answers; produce a clean, well-organized prompt in their voice's spirit."
- The output replaces or appends to `style_preferences.content`. If the user already has content, the UI must handle the merge gracefully (offer: "replace existing", "append to existing", "diff and let me pick").

### Guardrails

- Do not auto-commit the generated profile. The user must review and accept.
- Do not lock in a single phrasing — the user must always be able to edit the generated prompt before saving.
- Do not write to `exemplars` or `style_events` from the walkthrough. Those remain corpus-grown.
- Brief walkthrough = 5–8 questions, max. If you find yourself adding a 10th question to brief mode, move it to deep mode.

### Acceptance criteria

- A new user lands in onboarding and sees the three options (brief / deep / skip).
- Both walkthroughs run end-to-end and produce a draft voice profile the user can accept, edit, or discard.
- The walkthrough is re-runnable from the Journal → Style Guide panel.
- The drafted profile, once accepted, is the value of `style_preferences.content`.

## 5. Agent persona walkthrough (brief / deep)

### Goal

Give non-power users a guided way to create a custom **agent persona** (Editor, Critic, Author, etc.) without writing a prompt themselves. Same shape as the voice profile walkthrough, applied to agent creation.

### What ships

- When the user clicks "Create new agent" (in the Journal → Style Guide → Agents area, or wherever Agents Level 1 lives in feature 6), they get the same three options:
  - **Brief walkthrough** — short Q&A producing a usable agent persona prompt.
  - **Deep walkthrough** — longer, branching Q&A.
  - **Pick from a starter** — pick one of the pre-built agent personas (see feature 6).
  - **Write my own** — manual prompt editor.
- The walkthrough asks role-shaped questions: "What kind of feedback do you want this agent to give?", "How blunt should they be?", "What's their literary taste?", "Should they ever rewrite, or only comment?", "Are they meant for first-draft work or polish?".
- The deep walkthrough asks more nuanced questions: "Imagine you sent them a paragraph you secretly love but suspect is overwrought. What do you want them to say?".
- The end state — the drafted agent persona — gets reviewed, edited, and saved as a new row in `agents` (see feature 6 for schema).

### UX intent

- This walkthrough is more playful than the voice profile one — it's about defining a *character* with a perspective, not extracting the user's own voice.
- The walkthrough should explicitly include questions about: role, tone of feedback, scope of access, output channel (rewrites vs. comments — though in v1 this is constrained, see feature 6), and "what taste would they bring."
- The walkthrough must include a "test your agent" step before saving: the user provides a short paragraph, the freshly-defined agent runs on it, the user sees the output and can adjust the persona before committing.

### Architecture notes

- Same state-machine pattern as feature 4. The system prompt for the question-asking model is different: "you are helping a user define a custom editing/writing agent persona — ask the right questions to capture role, taste, tone, and constraints."
- The "compose final prompt" call produces the agent's `persona_prompt` field.
- The "test your agent" step calls the agent endpoint (feature 6) with the just-drafted persona — no commit.

### Guardrails

- Same as feature 4 (no auto-commit, user always reviews and edits).
- The walkthrough must not propose tools/access scopes that v1 doesn't support. (See feature 6 for what's available.)

### Acceptance criteria

- "Create new agent" flow offers walkthroughs (brief/deep), starters, or manual.
- Either walkthrough produces an agent persona the user can test before saving.
- Saved agents land in the `agents` table per feature 6's schema.

## 6. Agents Level 1 — custom editor/author agents composed with the voice profile

### Goal

Establish the **agent primitive** as a first-class entity. A v1 agent is a persona prompt that composes with the user's voice profile and runs against either a selection or a whole document, returning either a rewrite (committed via the existing strip flow) or a structured response rendered in a side panel. Margin comments and document-walking passes are out of scope for v1 (those land in v1.5).

### Calibration on what this is and isn't

The user raised a question about whether the "voice profile" in this concept is the user's voice or an agent's. To clarify:

- **The user's voice profile** is theirs, built from their preferences + exemplars + edit pairs. It expresses *how the user writes*.
- **An agent's persona** is a separate prompt the user authors (or generates via the walkthrough in feature 5). It expresses *how that agent thinks, critiques, edits, or writes*.
- An agent invocation composes both: the system prompt is roughly `(agent.persona_prompt) ⊕ (optional: user voice profile, depending on the agent's role) ⊕ (selection / doc context)`.

Some agents *want* to use the user's voice (a "Ghost Author" continuing the user's draft should write in the user's voice). Other agents *don't* (a "Cold Editor" critiquing the draft is meant to bring their own perspective, not echo the user's). The agent definition includes a flag for this.

### What ships

- A new `agents` table (illustrative — finalize naming during implementation):

  ```
  agents (
    id, user_id, name, role (text or enum: editor | author | critic | other),
    persona_prompt text,
    uses_voice_profile boolean default true,
    voice_profile_scope (nullable; if set, overrides default scope when composing),
    output_kind text ('rewrite' | 'response'), -- v1: these two only
    default_model text, default_temperature numeric,
    created_at, updated_at
  )
  ```
- An "Agents" management surface in Journal → Style Guide (or a dedicated tab inside the Style Guide panel — see feature 9). User can create, edit, delete, and test agents.
- Two ways to invoke an agent in v1:
  - **On a selection** (the user selects text and picks "Run with [agent]" from the bubble menu or command palette). If `output_kind = 'rewrite'`, the result lands in the Rewrite Strip. If `output_kind = 'response'`, it opens in a side panel.
  - **On a whole document** (via the doc-action button, similar to doc-level transforms). Same output split.
- A `/api/ai/agents/[id]/run` endpoint that composes the system prompt, runs the model with streaming, stamps `usage_events` with `transformId = "agent:<id>"`, and (for rewrite output) optionally produces `style_event` rows on commit.
- A small starter pack of pre-built agents available to clone-on-create (the user copies the persona into a new row they own — agents are per-user, no sharing in v1):
  - **Sentence Surgeon** (output_kind: rewrite, uses voice: true) — pure copy-edit.
  - **Cold Editor** (output_kind: response, uses voice: false) — comments only, ruthless cuts, asks "is this earning its place?".
  - **Generous Reader** (output_kind: response, uses voice: false) — finds what's working, encourages.
  - **Skeptic** (output_kind: response, uses voice: false) — only asks questions.
  - **Ghost Author** (output_kind: rewrite, uses voice: true) — continues the draft in voice.

### UX intent

- Agents should feel like *named collaborators* with consistent personalities. The list of agents should look like a small roster with avatars-or-equivalent, names, and a one-line description of their taste. Not like a settings list.
- Running an agent on a selection should feel as fast as the existing rewrite — same strip, just labelled with the agent's name.
- Running an agent on a doc with `output_kind = 'response'` opens a side panel showing the agent's structured response (probably markdown — bullets, callouts, suggestions). v1: this is a read-only side panel; the user can copy from it but can't directly apply changes from it. (v1.5 adds margin comments + doc passes, which is when responses become interactive.)
- The starter agents are clearly labelled as starters. Users should feel no friction in tweaking them.

### Architecture notes

- Schema: new `agents` table per the sketch above. Add a foreign key from `style_events` and `usage_events` to `agents.id` (nullable) so we can attribute events to the agent that produced them.
- Voice profile composition: the existing `compileVoiceProfile()` function returns `compiledSystemPrompt` and `cachedSystemBlocks`. For agents, the request's system prompt becomes `agent.persona_prompt + "\n\n" + compiledSystemPrompt` (when `uses_voice_profile = true`). Both blocks should be wrapped as cached system blocks for Anthropic prompt caching — the agent persona changes rarely, the voice profile changes occasionally, both are excellent cache candidates.
- Output rendering for `response` agents: a streaming markdown panel. Use a minimal markdown renderer that's safe (no arbitrary HTML).
- Hotkey allocation: don't claim more global hotkeys for agents in v1. Agents are picked via the bubble menu (selection) or the doc-action button (whole-doc) or the command palette.
- All agent-touching surfaces (creation, edit, run) must respect that the agent is per-user; RLS enforces this.

### Guardrails

- v1 agents do **not** post margin comments. They render in the strip (rewrite) or a side panel (response). Margin comments are v1.5.
- v1 agents do **not** auto-walk the document block by block. They run on the selection or the whole doc as a single call. Block-by-block passes are v1.5.
- v1 agents do **not** have memory beyond their persona prompt. Each run is stateless. Memory is v1.5/Future.
- v1 agents do **not** call tools. No web search, no doc-graph queries, no anything. (Web search exists as a separate capability via feature 7, but agents can't trigger it in v1.) Tool-using agents are Future.
- Don't build a panel of agents (running multiple at once) in v1. That's Agents Level 3 / v1.5.
- Cap response length sensibly to keep cost predictable. A "Cold Editor" running on a 5,000-word doc shouldn't return 5,000 words of feedback.

### Acceptance criteria

- The user can create, name, edit, test, and delete agents.
- The user can pick from the starter pack and clone any starter into their own agent.
- The user can run an agent on a selection and see the result in the existing strip flow (if rewrite-shaped) or in a side panel (if response-shaped).
- The user can run an agent on a whole doc with the same result split.
- `usage_events.transformId` is stamped with `agent:<id>`.
- For rewrite-shaped agents, accepted outputs produce `style_events` linked to the agent.

## 7. Web search capability layer (basic)

### Goal

Add a **provider-neutral web search capability** to lexi — a thin wrapper that lets a transform or (in v1.5+) an agent perform a web search and get back structured results. v1 ships the layer plus one internal use of it (an "ad-hoc research" affordance), so the layer is exercised before v1.5 wires the citation transform on top.

### What ships

- A new capability under `src/lib/ai/tools/websearch/`:
  - A unified `webSearch(query, options)` function that picks the best available provider-side web search tool (Anthropic's `web_search_*` server tool; OpenAI's web search via the Responses API, or whichever current name they're using at implementation time).
  - Returns a normalized shape: `{ results: [{ title, url, snippet, sourceDate? }, ...], queryUsed: string }`.
  - Honors the same BYOK / env-fallback resolution as the rest of the AI layer (web search is generally bundled into the same provider call, but we should still record an explicit `usage_events` row with a `transformId` like `tool:web_search`).
- A tiny user-facing entry point in v1: an "Ad-hoc research" command in the command palette that lets the user enter a query, see results, and copy snippets/links into the doc. This is mainly to dogfood the layer; the real consumers are v1.5's citation transform and v1.5's tool-using agent work.
- Settings entry in the AI provider section showing whether web search is available for the user's current provider.

### UX intent

- The layer is mostly invisible in v1. The "ad-hoc research" surface is intentionally minimal — a small modal with a query field and a results list.
- When a result is added to the doc (via "insert as link" or "insert as quote"), the link is wrapped with the URL and the source date is recorded inline (formatted unobtrusively).

### Architecture notes

- This is a *capability*, parallel to the provider abstraction. Don't muddy the `AIProvider` interface with it; create a separate `WebSearchProvider` interface and per-vendor implementations.
- Per-vendor implementations: read **current** Anthropic and OpenAI docs at implementation time. Both vendors have changed their tool names and shapes recently; do not rely on guesses. The MCP doc-fetcher servers are available (`mcp__34860b6f-...` and `mcp__bbd8998f-...`) — use them.
- Cost: web search calls cost more than plain completions and have their own per-search pricing on some providers. Stamp `usage_events` with the search-specific cost as best we can.
- Rate limit: web search calls should count against the same `checkAiRateLimit` check, but consider a separate sub-budget if needed.

### Guardrails

- Do not invoke web search implicitly from any v1 transform. v1 transforms are all pure prose transformations; none of them need the web. (Citation, the obvious consumer, is v1.5.)
- Do not store search results long-term in the database. Cache them in-memory during a session; the user is responsible for inserting what they want into the doc.
- Do not present search results without source URLs. Every result must carry a clickable, copy-able URL.

### Acceptance criteria

- A `webSearch()` function works against at least one provider (Anthropic or OpenAI), with a clean fallback if the configured provider doesn't support web search.
- The "Ad-hoc research" command palette entry runs a query and shows results.
- Results can be inserted into the doc as links or quotes.
- A `usage_events` row is recorded for each web search call.

## 8. Auto-extracted preferences (manual "Analyze my edits" trigger)

### Goal

Close the loop between the user's accumulated `style_events` corpus and their declarative `style_preferences`. v1 ships a manual button that runs a one-shot analysis of recent edits and proposes new preference bullets for the user to accept/reject. v1.5 turns this into a periodic background job.

### What ships

- A button in the Journal → Style Guide panel (and from the existing voice profile preview area) labeled something like "Analyze my edits → suggest preferences."
- Clicking it runs an LLM call with a curated batch of recent `style_events` (~30–50 most relevant accepted rewrites with their before/after context) and asks the model to propose 3–7 candidate preference bullets, each with: a one-line statement, a one-line rationale, and 2 example pairs from the corpus that support it.
- The candidates are shown to the user in a review queue. Each can be: accepted (appended to `style_preferences.content`), edited then accepted, or rejected.
- Accepted candidates are appended with light formatting; the user can always go re-edit the prompt later.

### UX intent

- This should feel like the app *learning from the user* and *asking for confirmation*, not like the app deciding for them. Every candidate is a proposal, never an automatic write.
- Each candidate is presented with the supporting examples. "I noticed you almost always cut hedges like 'I think' and 'somewhat' — want me to add 'avoid hedging openers' to your preferences? [examples: ...]"
- This is also where the user discovers patterns they didn't know they had. Surface that delight.

### Architecture notes

- A new endpoint, e.g. `/api/style/analyze-edits`. Authoring of the analysis prompt matters — it should explicitly ask for *patterns the user is consistently making*, not single-instance edits.
- Input: ~30–50 recent `style_events` of type `rewrite` and `ai_suggestion_accepted`/`edited`, scoped to the user, with surrounding context.
- Output: structured candidates (use a model that can return structured JSON, or parse markdown carefully).
- The user-facing review queue can be a simple list with accept/edit/reject actions.

### Guardrails

- **Never auto-write to `style_preferences.content`.** Every change is user-confirmed.
- Don't over-propose. If the model returns 12 candidates, cap at the top 5–7 and label the rest as "more available."
- Don't propose candidates that are already in the user's preferences — diff against existing content first.
- Respect the user's tone in the proposals. "Avoid hedging openers" feels different from "stop being so wishy-washy"; the former is right.

### Acceptance criteria

- The "Analyze my edits" button works and produces 3–7 candidates with supporting examples.
- The user can accept, edit, or reject each candidate; accepted candidates land in `style_preferences.content`.
- Running with insufficient corpus (e.g. <5 edits) produces a friendly "not enough material yet" message instead of a model call.

### Why this is split (v1 vs v1.5)

The manual trigger pattern in v1 nails the analysis prompt, the candidate-review UX, and the `style_preferences` write path. v1.5 wraps that in a periodic background job (weekly cadence by default) and adds smarter pattern detection (clustering, deduplication against prior accepted/rejected candidates, suggesting deletions of stale preferences). Building the job machinery before the analysis logic is well-tuned would risk shipping a noisy notification feature.

## 9. Journal surface — top-level information architecture restructure

### Goal

Establish the **Journal** as a permanent top-level surface in lexi — the home for everything that exists at a meta-level above working documents. The journal has four panels; v1 stands all four up, with two fully functional and two minimal but real.

### What ships

- A new top-level navigation entry: **Journal**. Lives alongside the existing workspace (writing) surface.
- Replaces the existing `/style` route with `/journal`, which contains four panels (tabs or sub-routes — your call, but tabs read more cleanly):
  - **Style Guide** — view-only-for-now display of all training material: preferences, exemplars, recent rewrite pairs, voice profile preview, and (new in v1) the auto-extracted-preferences review queue. Includes the entry points for the voice profile walkthrough (feature 4) and the agent persona walkthrough + agent management (features 5, 6).
  - **Reflections** — placeholder in v1 with a small "Coming in v1.5" treatment AND one or two minimal stats (total docs, total words written this month, total `style_events` captured). Designed as a placeholder users will see and look forward to.
  - **Notes** — a basic markdown notepad surface where users can drop meta-level thoughts ("I've been writing too much in passive voice lately", "remember to use shorter paragraphs in the work-doc context"). v1 treats this as straightforward notes; future iterations may opt-in to using these notes to inform the voice profile or agents.
  - **Context** — surfaces all the user-supplied context: exemplars (with search and edit), and a placeholder area for "Sources" where future work will let the user upload files / paste URLs as reference material.

### UX intent

- Journal is calm. It's the "home base" for the user's writing identity, not a notification center.
- Style Guide is read-mostly. Editing is mediated — the user can edit `style_preferences.content` directly (existing behavior), but there's a clear note that in the future, edits to compiled training material will be model-mediated (see Future plan: "Style guide model-mediated editing"). v1 keeps direct editing as is.
- Reflections in v1 is a deliberate teaser. It must not look broken; the placeholder state should communicate "this is real, it's coming, and here's a glimpse." Two or three live stats are enough.
- Notes is plain. A markdown editor (we can reuse a simplified TipTap setup), per-user, single-doc-style. No projects, no fancy structure in v1.
- Context starts as an exemplar viewer and a placeholder for sources.

### Architecture notes

- Move the existing `/style` page into `/journal/style-guide`. Keep the URL `/style` redirecting to `/journal/style-guide` for any existing bookmarks.
- Tabs: `/journal/style-guide`, `/journal/reflections`, `/journal/notes`, `/journal/context`. Or a single `/journal` page with state-managed tabs — either is fine; routes are slightly nicer for deep-linking.
- New table: `notes` (id, user_id, content text, updated_at). One row per user in v1; multi-note structure is a Future concern.
- For Reflections stats: read from existing tables. `documents.count`, `style_events.count`, and a `sum(word_count)` over docs updated this month is enough.

### Guardrails

- Do not move existing settings into Journal. Settings stay at `/settings`. Journal is for the user's writing identity / context, not their app config.
- Do not over-promise on Reflections. The placeholder must feel intentional, not unfinished.
- Do not couple Notes to other systems in v1. It's just notes.

### Acceptance criteria

- `/journal` exists and shows four tabs: Style Guide, Reflections, Notes, Context.
- Style Guide contains all content from the existing `/style` page plus walkthrough entry points and the auto-extract feature.
- Reflections shows 2–3 live stats and a "more coming" treatment.
- Notes lets the user write and save markdown notes.
- Context shows exemplars (via the existing list) plus a sources placeholder.
- `/style` redirects to `/journal/style-guide`.

## 10. Encrypt `user_credentials.api_key` (security baseline)

### Goal

The MVP stores BYOK API keys in plaintext. Before any of the v1 work expands the surface area of credentials handling (more agents, more transforms, more cost), encrypt at rest. This is on the existing roadmap and is small enough to land cleanly in v1.

### What ships

- A KMS-backed (or libsodium / age / similar — pick one well-supported library appropriate for the deployment target) encryption layer for `user_credentials.api_key`.
- A migration that re-encrypts existing rows once the encryption key is configured.
- A documented key-rotation procedure in the README.

### UX intent

Invisible to the user. Settings continue to look the same.

### Architecture notes

- Decide: server-side symmetric encryption with a key from env (simplest, weakest), or KMS-backed (best, requires deploy-environment KMS — Vercel doesn't ship one natively but the deploy target may). A defensible v1 choice is symmetric server-side with `LEXI_CREDENTIALS_KEY` env var; rotation is documented.
- All read paths (`credentialsRepo.getDefaultForUser`, list, etc.) decrypt on read. All write paths encrypt on write.
- Make sure the migration is idempotent — running it twice must not double-encrypt.

### Guardrails

- Do not log decrypted keys. Do not return them in API responses (the existing repo already only returns label + provider; keep it that way).
- Do not break BYOK if the env key is missing — fail loudly so the operator sets it before any user tries to use BYOK.

### Acceptance criteria

- New BYOK keys are encrypted at rest.
- Existing keys are migrated to encrypted form on first deploy with `LEXI_CREDENTIALS_KEY` set.
- AI calls work end-to-end with encrypted keys.

## Sequencing recommendation for v1

The ten features above don't have to ship in numbered order; some are strongly parallelizable and some depend on others.

**Foundation wave (do first, parallelizable):**
- Feature 1 (streaming + variants) — touches existing rewrite path; isolated.
- Feature 7 (web search capability) — isolated, no dependencies.
- Feature 10 (credential encryption) — isolated.

**Surface wave (do after foundation, can be parallel with each other):**
- Feature 2 (in-line transform shelf) — depends on feature 1's streaming path.
- Feature 3 (doc-level transforms + SoC → Draft) — depends on feature 1's streaming path. **This is the headline feature; give it the most care.**
- Feature 9 (Journal surface) — independent of features 1–8 except that features 4, 5, 6, 8 all surface entry points inside Journal → Style Guide; landing Journal first gives them a home.

**Identity wave (do after Journal exists, can be parallel with each other):**
- Feature 4 (voice profile walkthrough)
- Feature 5 (agent persona walkthrough)
- Feature 6 (Agents Level 1) — feature 5 is more useful once feature 6 exists, so 6 first or in parallel.
- Feature 8 (auto-extracted preferences) — independent; can be done any time once Journal has its Style Guide panel.

A reasonable PR ordering for one or two coding agents working sequentially:

1. Encrypt credentials (10).
2. Streaming + variants (1).
3. In-line transform shelf (2).
4. Doc-level transforms infrastructure + SoC → Draft (3).
5. Journal surface restructure (9).
6. Voice profile walkthrough (4).
7. Agents Level 1 (6).
8. Agent persona walkthrough (5).
9. Auto-extracted preferences (8).
10. Web search capability layer (7) — can also slot in earlier if there's bandwidth.
